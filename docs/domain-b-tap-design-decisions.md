## 0. 배경 — 도메인 구현 현황 스냅샷

논의 시작 시점 기준, B 도메인(출금·광고와 부스터·포인트·탭·친구초대) 중 실제로 동작하는 API는 없고 대부분 **엔티티/리포지토리만 존재**했다 (`TapBatch`, `UserTapDaily`, `PointAccount`, `PointLedger`, `CashoutRequest`, `BoosterGrant`). 친구 초대는 엔티티조차 없음. 서비스 로직을 짜기 전에 PRD가 열어둔 정책 상충(§13)들을 먼저 정리해야 한다는 문제의식에서 이 논의가 시작됨.

## 1. 탭(Tap) 도메인이 여러 도메인의 공용 원천 데이터라는 확인

**결론**: 탭 기록은 "키캡 상자용"이 아니라 최소 3개 도메인이 공유하는 데이터다.

| 소비처 | 근거 |
|---|---|
| 포인트(P) 적립 | PRD §7.1 — 300탭(둔화 시 600탭)당 1P |
| 키캡 상자 드롭 | PRD §7.2 — 평균 200탭당 1개, 목표 탭 수 사전 결정 |
| 랭킹 점수 | PRD §7.7 — "점수 = 유효 탭 수(어뷰징 필터 적용 후)" — **P가 아니라 탭 수 자체가 점수** |

**왜 중요한가**: 어뷰징 필터를 잘못 설계하면 P뿐 아니라 랭킹 순위까지 왜곡된다. 탭 검증 로직을 "상자 하나만을 위한 것"으로 좁게 설계하면 안 된다는 근거.

## 2. 무효 탭 처리 — 롤백 전략 (PRD §13-4 해소)

**결정**: 서버는 검증에서 무효로 판정된 탭을 조용히 반려(집계에서 제외)하고, 프론트는 별도 롤백 UI 없이 이미 보여준 낙관적 카운트를 그대로 둔다. 다음 배치/동기화 시점에 서버 확정값으로 자연스럽게 수렴시킨다.

**이유**: 탭 카운트 하나하나에 대해 "이 탭은 무효였으니 화면에서 빼주세요" 같은 정밀 롤백 UX를 만드는 비용이, 어차피 P·랭킹은 서버 확정값 기준이라는 원칙(PRD §7.1 "낙관적 UI + 서버 확정")과 비교해 실익이 낮다고 판단.

---

## 3. 일반 적립 곡선 수치 — 150 vs 300 (PRD §13-15 해소)

**결정**: 300탭당 1P로 확정. 150 폐기.

**근거**: PRD 원문 내 상충 — 일부 문서는 "온보딩 150 / 일반 300 / 둔화 600"(SPEC 02), 다른 곳은 "온보딩 이후 150탭당 1P"(온보딩 SPEC 1)로 기재. 온보딩 실측 자체가 15탭=1P이므로 150이라는 값이 실제로 쓰이는 곳이 없다는 점, 클릭 정책 카드·리워드 카드가 300을 채택하고 있다는 점 근거로 300으로 확정. Figma 프레임(`65:2`)에서도 "온보딩 15탭 / 일반 300탭 / 둔화 600탭당 1P"로 재확인.

## 4. 적립 곡선 분포 형태 (PRD §13-5 해소)

**결정**: 균등분포(Uniform) 채택.

**후보 비교**:
| 방식 | 특징 |
|---|---|
| 균등분포 (채택) | 구현 단순. 범위 내 모든 값이 동일 확률 |
| 정규분포 | 300 근처에 확률이 몰려 직관과 맞지만 구현이 더 복잡, 극단값 클리핑 필요 |

**변동폭 확정**: 곡선 단계별로 다르게 적용한다.
- 일반 곡선(300 기준): `r = ±10%` → 목표 범위 270~330
- 둔화 곡선(600 기준): `r = ±5%` → 목표 범위 570~630

`AppConfig`에 곡선 단계별 변동폭을 키로 저장해 무배포로 조정 가능하게 한다.

## 5. 목표치 산출 메커니즘 — "사전 결정형"

**결정(잠정)**: 키캡 상자 드롭(PRD §7.2)과 동일한 패턴을 포인트에도 적용. 직전 P 지급 시점에 서버가 **다음 목표 탭 수를 미리 균등분포로 추첨해서 저장**하고, 이후 배치가 올 때마다 "누적 유효 탭 ≥ 저장된 목표"만 비교한다.

**대안이었던 방식**: 사후 계산형(탭마다 확률을 굴려 즉시 지급 여부 판정) — 구현이 더 복잡하고 상자 드롭과 로직이 불일치하게 되어 기각.

**목표치 노출 여부**: 서버는 "다음 P까지 남은 탭 수"를 프론트에 내려주지 않기로 함. 이유: 이 값을 노출하면 "예측 방지"라는 곡선 설계 의도(PRD §7.1 "예측 방지") 자체가 무력화됨. 트레이드오프로 P 반영이 최대 30초(배치 주기) 늦어질 수 있는 점은 감수.

## 6. 리워드 산정 기준 — 세션 단위가 아니라 일일 누적

**확인 사실**: `TapBatch.tapSessionId` + `sequence`는 **멱등성/중복 전송 방지 용도**일 뿐이고, 실제 리워드(P·둔화 판정·일일 상한)는 `UserTapDaily`(일 단위 집계 테이블) 기준이다. 앱을 껐다 켜서 세션이 바뀌어도 당일 누적치는 이어진다.

**왜 처음에 헷갈렸는지**: `TapBatch`에 세션 개념이 스키마로 존재해서 "세션 종료 시 정산"으로 오해하기 쉬웠음. 실제로는 세션은 배치 순서 보장/중복 방지 계층이고, 리워드 계층(`UserTapDaily`)과 분리되어 있음.

## 7. 배치 처리 아키텍처 — 프론트 전송 전략과 백엔드의 분리

**결정**: 백엔드는 프론트가 "왜"(30초 경과/100탭 도달/조기 flush 등) 배치를 보냈는지 알 필요가 없다. 배치가 오면 항상 동일한 로직(검증 → 누적 → 목표 비교 → 지급 → 응답)만 수행한다.

**초기 오해였다가 정정된 부분**: "한 배치 안에서 목표를 2번 이상 넘는" 엣지 케이스를 방어 로직으로 준비해야 한다고 처음 제안했으나, 배치 최대 크기(100탭)와 목표치 평균(300, 상식적 변동폭 기준 최소값도 100 초과)을 비교하면 **실무적으로 발생 불가능**하다는 지적을 받고 정정함. 변동폭을 극단적으로 넓게 잡지 않는 한 특수 처리는 불필요.

## 8. 배치 엔드포인트 Rate-limit

**문제의식**: 탭 유효성 검증(80ms 간격, 분당 420회 등)과는 별개로, **배치 요청 자체의 남발**(예: 작은 배치를 초당 여러 번 전송)이 DoS 벡터가 될 수 있다는 점을 확인. PRD §7.6은 "탭의 유효성"만 다루고 "배치 요청 빈도" 자체에 대한 통제는 명시하지 않음 — 발견한 갭.

**후보 비교**:
| 방식 | 설명 | 트레이드오프 |
|---|---|---|
| 고정 최소 간격 | 마지막 배치 후 N초 이내 재요청 거부 (N ≈ 8초, 100탭×80ms 물리적 최소치에서 역산) | 구현 단순 / 정상적인 재시도·조기 flush까지 오탐 가능 |
| 토큰버킷 (분당 N회) | 용량(예: 8개)만큼 burst 허용, 이후 일정 속도로만 재충전 | burst 허용으로 정상 케이스 흡수 / 구현·상태 관리(Redis) 복잡도 ↑ |

**결정**: 토큰버킷 방식 채택. 근거: PRD §7.6이 명시한 "오프라인 로컬 누적 → 복귀 시 재검증" 시나리오가 여러 배치를 한꺼번에 재전송하는 정상적인 burst 케이스이기 때문에, 이를 예외 처리 없이 자연스럽게 흡수할 수 있는 토큰버킷이 고정 최소 간격보다 적합하다고 판단.

**남은 미결**: 구체적인 용량/충전 속도 수치는 미정 (예시로 든 "8개, 8초당 1개"는 검증되지 않은 추정치 — 구현 시 초기값으로 시작해 튜닝).

## 9. 봇 탐지 — 표준편차 임계값 (PRD §7.6)

**리서치 배경**: 서비스가 아직 런칭 전이라 자사 데이터가 없어, 업계 일반 연구/벤치마크로 시작값을 추정.

**리서치 결과 요약**:
- 사람의 고속 버스트 클릭(40~65ms 간격대)은 표준편차 약 8ms 수준
- 고정 간격 오토클리커(예: 50ms 고정)는 표준편차 1ms 미만
- 더 느슨한 일반 클릭 컨텍스트에서는 사람 약 59ms, 자동화 도구 5ms 미만이라는 자료도 있음
- 게임 봇 탐지 학술 연구(ETRI Journal, 2023)도 "행동 간 시간 간격 평균+표준편차"를 탐지 피처로 사용

**결정**: 꾹머니는 최소 간격 80ms 기준의 고속 연타 컨텍스트이므로, **표준편차 10~15ms 미만이면 봇 의심**으로 잠정 시작값을 채택. 런칭 후 실제 유저 클릭 로그로 재조정하는 것을 전제로 한 임시값.

## 10. 클릭 간격(규칙성) 데이터 수집 방식 — 가장 오래 논의된 지점

### 10.1 원본 PRD/Figma가 실제로 뭐라고 했는지 확인

Figma 프레임 `46:6`("명세 — 어뷰징 정책")을 직접 열어 확인한 결과, PRD.md §7.6은 이 프레임 텍스트를 그대로 옮긴 것이었다:

> `46:17` — "30초·100탭 배치 전송 → 서버가 유효탭 재계산 + 규칙성 검사 (클라 불신뢰)"

**"클라 불신뢰" 문구의 정확한 해석**: 처음엔 이 문구가 "규칙성 검사"까지 포함해 전체를 불신한다는 뜻으로 읽었으나, 문장을 분해해보면 **"서버가 유효탭 재계산"(= 클라가 주장하는 탭 개수·P를 그대로 안 믿고 서버가 자체 룰로 재계산)에 붙는 근거**로 보는 게 더 타당하다고 정정함. "재계산"은 서버가 이미 가진 데이터(배치 도착 시각+신고 개수)만으로 가능하지만, "규칙성 검사"에 쓰일 원본 데이터(클릭 간격)의 신뢰 모델은 PRD/Figma 어디에도 명시돼 있지 않다는 것이 최종 결론. **즉 이 설계 질문은 PRD/Figma가 답을 갖고 있지 않은, 백엔드가 새로 정해야 하는 영역**임을 원본 확인으로 재확인.

### 10.2 검토한 옵션들

| 옵션 | 데이터 출처 | 신뢰도 | 정밀도 | 기존 배치 스펙(100탭/30초) 변경 여부 |
|---|---|---|---|---|
| **① 배치 도착 시각만 사용 (최종 채택)** | 서버 자체 시계 | 100% (조작 불가) | 낮음 (배치 단위, 8~30초 해상도) | 변경 없음 |
| ② 클릭 타임스탬프 배열 포함 ("하이브리드"/PRD 원안에 가까움) | 클라이언트가 기록해서 전송 | 낮음 (조작 가능 — 봇이 배열을 지어낼 수 있음) | 높음 (PRD 스펙 "100클릭 표준편차" 그대로 충족) | 변경 없음(페이로드 필드만 추가) |
| ③ 서버 강제형 소단위 배치 (예: 20탭 캡) | 서버 자체 시계 | 100% | 중간 (20탭 단위 근사) | **변경 있음** — 기존 100탭/30초 계약을 재협의해야 함 |
| ④ 실시간 개별 탭 전송 | 서버 자체 시계 | 100% | 가장 높음 (진짜 클릭 단위) | **변경 있음** — 배치 개념 자체를 대체, 요청량 폭증 |
| ⑤ 표준편차 지표 자체 폐기 | 서버 소유 지표만 (최소간격/분당상한/일일상한) | 100% | 규칙성 검사 자체를 포기 | PRD 요구사항 일부 미충족 |

### 10.3 최종 결정 — ① 채택

**결정**: 클릭 타임스탬프는 클라이언트로부터 받지 않는다. 기존 `TapBatch.createdAt`(서버 수신 시각) 간의 간격만으로 규칙성을 근사 판정한다.

**선택 이유**:
1. 기존 100탭/30초 배치 스펙(PRD/Figma 문서화됨, 프론트와의 계약)을 건드리지 않아도 됨 — 프론트 재작업·기획 재협의 불필요
2. 서버가 전혀 조작 불가능한 데이터만 신뢰 — "클라 불신뢰" 원칙에 가장 부합
3. 완전히 기계적인 봇(고정 간격 클릭)은 배치를 채우는 데 걸리는 시간도 기계적으로 일정해지므로, 배치 단위 근사치로도 가장 노골적인 봇은 충분히 탐지 가능

**받아들인 트레이드오프**: PRD 문구가 말한 "최근 100클릭 간격 표준편차"를 문자 그대로는 구현하지 못하고, 배치 단위(8~30초 해상도) 근사치로 대체함. 정교하게 설계된 봇(배치 단위로는 자연스럽게 보이도록 흐름을 조절하는 경우)은 놓칠 수 있음 — 이후 데이터가 쌓이면 재검토 대상.

**기각한 이유**:
- ②(클릭 타임스탬프): 조작 가능성이 근본적 — 봇 스크립트를 짜는 사람이 배열 값도 함께 조작할 수 있어 검증 재료로서 신뢰 불가
- ③(서버 강제형 소단위 배치): 이미 문서화된 API 계약(100탭/30초)을 깨는 변경이라 프론트/기획과의 재협의 비용이 큼
- ④(실시간 전송): 배터리·네트워크 부담 및 요청량 폭증, rate-limit 설계도 훨씬 정교하게 다시 필요

## 11. 둔화(600탭) 전환 처리

**확인 사실**: PRD §7.1 "둔화 | **하루 7P 이상 획득 시** 둔화 진입 → 약 600탭당 1P" — 트리거 조건은 탭 수가 아니라 **그날 누적 P**다.

**도출된 결론**: 목표치 재추첨은 "P가 지급되는 순간"에만 일어나고, 둔화 트리거(하루 누적 P ≥ 7)도 오직 그 순간에만 값이 바뀔 수 있다. 즉 "이미 진행 중인 카운트다운을 중간에 끊어야 하는" 상황 자체가 구조적으로 발생하지 않는다 — 목표치를 새로 뽑을 때마다 "그 순간 오늘 누적 P가 7 이상인지"만 확인해서 300/600 기준을 선택하면 된다. (처음엔 "이어가기 vs 즉시 재계산" 두 가지 선택지가 있다고 봤으나, 트리거 조건이 P 기반이라는 점을 재확인하면서 이 이분법 자체가 불필요하다는 것으로 정정됨.)

**남은 디테일**: 한 배치 안에서 목표를 여러 번 연속으로 넘는 극단적 경우(§7 항목 참고, 현재는 실무상 거의 불가능하다고 판단)를 대비해, 목표 도달 판정 로직을 while 루프 형태로 짜두면 안전.
