# 꾹머니 13개 테이블 PostgreSQL 명세

> 사용자 PK 기준: `app_user.id UUID`, 모든 `user_id UUID`

이 문서는 1차 Persistence MVP의 테이블, 컬럼, 제약, 인덱스 Source of Truth다. 실제 DB 구조는 Flyway Migration 기준으로 문서화한다.

## 공통 규칙

- `app_user.id`는 `UUID` PK이며 Java `GenerationType.UUID`로 생성한다.
- `app_user`에는 별도 `public_id`를 두지 않는다. 사용자 API ID, JWT `sub`, Redis 사용자 식별자, 모든 `user_id` FK는 같은 UUID를 사용한다.
- 나머지 테이블 PK는 `BIGINT GENERATED BY DEFAULT AS IDENTITY`를 사용한다.
- 나머지 리소스의 `public_id`는 UUID이며 API에서 직접 식별할 필요가 있을 때 사용한다.
- 시간은 `TIMESTAMPTZ`로 저장한다.
- FK의 기본 삭제 정책은 `ON DELETE RESTRICT`다.
- 탈퇴 시 금액성·원장 데이터를 물리 삭제하지 않으므로 사용자 FK도 유지한다.
- 상태 문자열은 `VARCHAR`와 CHECK 제약을 사용한다.
- 수량과 잔액은 0 미만이 되지 않도록 CHECK와 서비스 검증을 함께 둔다.
- 잔액과 수량 변경 테이블에는 `version`을 두고 낙관적 잠금 또는 명시적 행 잠금을 사용한다.
- Toss Access Token, Toss Refresh Token, authorizationCode는 어떤 테이블에도 저장하지 않는다.

## 1. app_user

빵도감의 `User.id UUID` 전략을 참고해 사용자 PK 자체를 API 식별자로 사용한다.

| 컬럼 | 타입 | NULL | 기본값 | 제약과 설명 |
|---|---|---:|---|---|
| `id` | UUID | N | Java UUID 생성 | PK, 사용자 API ID, JWT `sub` |
| `status` | VARCHAR(20) | N | `ACTIVE` | CHECK `ACTIVE`, `SUSPENDED`, `WITHDRAWN` |
| `nickname` | VARCHAR(50) | Y | | 표시 닉네임. 탈퇴 시 익명화 |
| `nickname_normalized` | VARCHAR(50) | Y | | 닉네임 중복 검사 값. 탈퇴 시 null 또는 익명값 |
| `profile_image_url` | TEXT | Y | | Toss 프로필 이미지. 탈퇴 시 null |
| `onboarding_reward_claimed` | BOOLEAN | N | false | 온보딩 보상 중복 지급 방지 |
| `onboarding_completed_at` | TIMESTAMPTZ | Y | | 온보딩 완료 시각 |
| `last_login_at` | TIMESTAMPTZ | Y | | 마지막 로그인 |
| `withdrawn_at` | TIMESTAMPTZ | Y | | 회원 탈퇴 처리 시각 |
| `created_at` | TIMESTAMPTZ | N | now() | 생성 시각 |
| `updated_at` | TIMESTAMPTZ | N | now() | 수정 시각 |

인덱스와 제약:

- `ux_app_user_active_nickname_normalized ON app_user(nickname_normalized) WHERE nickname_normalized IS NOT NULL AND status='ACTIVE'`
- `ix_app_user_status(status)`
- `CHECK ((status='WITHDRAWN' AND withdrawn_at IS NOT NULL) OR status<>'WITHDRAWN')`

탈퇴 정책:

- 행을 물리 삭제하지 않는다.
- `status=WITHDRAWN`, `withdrawn_at=now()`로 변경한다.
- 닉네임과 프로필 이미지를 익명화한다.
- 포인트 원장, 출금, 상자 개봉 기록의 FK는 유지한다.

## 2. auth_identity

| 컬럼 | 타입 | NULL | 기본값 | 제약과 설명 |
|---|---|---:|---|---|
| `id` | BIGINT | N | identity | PK |
| `public_id` | UUID | N | Java 생성 | UNIQUE |
| `user_id` | UUID | N | | FK `app_user(id)` |
| `provider` | VARCHAR(30) | N | | CHECK `TOSS` |
| `provider_user_id` | VARCHAR(255) | N | | Toss `userKey` 문자열. 로그 출력 금지 |
| `linked_at` | TIMESTAMPTZ | N | now() | 연결 시각 |
| `created_at` | TIMESTAMPTZ | N | now() | 생성 시각 |
| `updated_at` | TIMESTAMPTZ | N | now() | 수정 시각 |

Unique:

- `(provider, provider_user_id)`
- `(user_id, provider)`

정책:

- `provider_user_id`는 `String.valueOf(login-me.userKey)`다.
- Toss Access/Refresh Token은 이 테이블이나 `app_user`에 저장하지 않는다.
- 탈퇴 사용자 Identity는 Webhook 멱등성, 중복 온보딩 보상 방지, 자동 재가입 차단을 위해 MVP 보관 정책 동안 유지한다.
- `app_user.status=WITHDRAWN`인 Identity 로그인은 `ACCOUNT_WITHDRAWN`으로 거절한다.

## 5. user_keycap

| 컬럼 | 타입 | NULL | 기본값 | 제약과 설명 |
|---|---|---:|---|---|
| `id` | BIGINT | N | identity | PK |
| `public_id` | UUID | N | Java 생성 | UNIQUE |
| `user_id` | UUID | N | | FK `app_user(id)` |
| `keycap_id` | BIGINT | N | | FK `keycap(id)` |
| `shard_count` | INTEGER | N | 0 | CHECK `>= 0` |
| `status` | VARCHAR(20) | N | `IN_PROGRESS` | CHECK `IN_PROGRESS`, `COMPLETED` |
| `equipped` | BOOLEAN | N | false | 현재 장착 여부 |
| `completed_at` | TIMESTAMPTZ | Y | | 완성 시각 |
| `created_at` | TIMESTAMPTZ | N | now() | 생성 시각 |
| `updated_at` | TIMESTAMPTZ | N | now() | 수정 시각 |

Unique와 인덱스:

- `(user_id, keycap_id)`
- `UNIQUE (user_id) WHERE equipped=true`
- `ix_user_keycap_user_status(user_id, status)`

온보딩 완성 키캡은 `shard_count=required_shard_count`, `status=COMPLETED`로 직접 지급한다.

키캡 장착 API 구현은 `status=COMPLETED`인 사용자 보유 키캡만 `equipped=true`로 변경한다. 같은 사용자의 기존 장착 키캡은 같은 트랜잭션에서 `equipped=false`로 해제하며, 사용자당 `equipped=true` 최대 1개 정책은 위 partial unique index를 최종 DB 제약으로 사용한다.

## 6. keycap_box_account

| 컬럼 | 타입 | NULL | 기본값 | 제약과 설명 |
|---|---|---:|---|---|
| `id` | BIGINT | N | identity | PK |
| `public_id` | UUID | N | Java 생성 | UNIQUE |
| `user_id` | UUID | N | | FK, UNIQUE `app_user(id)` |
| `box_balance` | INTEGER | N | 0 | CHECK `>= 0` |

| `created_at` | TIMESTAMPTZ | N | now() | 생성 시각 |

현재 구현에서 상자 진행도는 `keycap_box_account`가 아니라 `user_tap_progress.cumulative_valid_tap_count`, `user_tap_progress.next_box_target`을 원본으로 사용한다.

아래 컬럼은 무료권 충전과 광고 개봉 정책이 확정될 때 후속 설계로 분리한다. 현재 Entity와 조회 API 구현 완료 컬럼으로 보지 않는다.

| 후속 설계 후보 컬럼 | 타입 | 설명 |
|---|---|---|
| `next_free_ticket_at` | TIMESTAMPTZ | 다음 무료권 충전 시각 |
| `ad_open_date` | DATE | 광고 개봉 카운트 기준일 |
| `ad_open_count` | INTEGER | 광고 개봉 횟수 |
| `box_progress_tap_count` | INTEGER | `user_tap_progress`와 중복 저장 여부 결정 필요 |
| `next_box_required_tap_count` | INTEGER | `user_tap_progress.next_box_target`과 중복 저장 여부 결정 필요 |

## Migration과 Entity 정합성 검토

- 현재 문서의 컬럼, 제약, 인덱스는 `V1010__create_a_domain_schema.sql`과 `V1020__drop_auth_session_log.sql` 기준이다.
- Entity는 13개 테이블명, PK 타입, 주요 FK 타입, `public_id` 보유 여부에서 Migration과 맞는다.
- Migration 기준으로 `point_ledger.user_id`와 `point_account_id`가 같은 사용자임을 보장하는 DB 제약은 없다.
- Migration 기준으로 `booster_grant`의 사용자별 활성 부스터 1개 제한은 없다.
- Migration 기준으로 `user_keycap.equipped=true`가 `status=COMPLETED`일 때만 가능하다는 CHECK 제약은 없다.
- 키캡 장착 API는 서비스 레벨에서 완료 상태를 검증하고 사용자 키캡 행 잠금 후 상태를 변경한다. 실제 공유/개발 DB의 `UNIQUE (user_id) WHERE equipped=true` 존재 여부는 배포 전 확인이 필요하다.
- `app_config`는 `(config_key, effective_at)` Unique와 인덱스를 갖고, 현재 Repository에는 유효 시각 기준 최신 설정을 조회하는 `findFirstByConfigKeyAndEffectiveAtLessThanEqualOrderByEffectiveAtDesc(...)`가 있다.
